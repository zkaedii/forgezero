## v0.1.1 Bug: KI Parser False-Positive

**Finding:** The KI report shows `brain [DETECTED_WHEN_EXPLICIT]` at Step 105. `brain` is not a KI.

**Root cause:** The KI parser extracts path segments from `overview.txt` that contain `/knowledge/<slug>/`, but the path at Step 105 was `~/.gemini/antigravity/brain/<conv-id>/...` — the parser erroneously matched `brain` as a slug.

**Verified:** `ls ~/.gemini/antigravity/knowledge/` shows exactly three KI directories:
- `zcc_doom_milestone`
- `zcc_ir_bridge`
- `zcc_ir_bridge_pipeline_state`

No `brain` KI exists. The detection is a false positive.

**Fix (v0.1.1):** KI path extraction must validate extracted slugs against the actual directory listing of `~/.gemini/antigravity/knowledge/` before reporting. Any slug not found on disk should be dropped silently or logged as `[UNRESOLVED]`.

## v0.2.0 Design Constraint

**The constraint that must survive the build:**

> The tool must never lie about what it can see.

v0.1.0 optimized false-positive-zero on detection, with the honest cost (lower-bound reporting) made explicit in every output. `forge0 trace` must extend detection coverage *without* degrading this property.

The confidence ladder for v0.2.0:

| Tag | Meaning | Version |
|-----|---------|---------|
| `[DETECTED_WHEN_EXPLICIT]` | Confirmed via `view_file` tool call in logs | v0.1.0 |
| `[DETECTED_VIA_TRACE]` | Confirmed via trace mechanism (process event, session log, system prompt capture) | v0.2.0 |
| `[AVAILABLE_UNCONFIRMED]` | Skill was in the available list; load not confirmed | v0.2.0 |

Build the confidence ladder before building the trace mechanism. The moat is not tracing. The moat is tracing *while preserving the property that the tool never lies about what it can see.*

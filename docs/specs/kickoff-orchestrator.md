## 1. Thesis & Polarity

The `forge0 kickoff` command creates a session dump at the beginning of an agent interaction to provide a verifiable snapshot of the environment. The risk classification of the session determines the depth of the dump ("mode").

The critical distinction defining mode intent:

> **Minimal kickoff preserves traceability of *who and when*.**  
> **Full kickoff preserves traceability of *what was visible*.**  
> **Auto selects based on what the session looks like it's about to do.**

### 1.1 Inverted Polarity (Fail-Closed)

Mode selection must be derived entirely from independent observables, never solely from the agent's self-reported intent. Agent intent is susceptible to a **downgrade attack** (e.g. adversarial framing of a security rewrite as a "typo fix").

Instead of a denylist of dangerous keywords, `auto` mode starts at `full` and only downgrades to `minimal` when **all** affirmative low-risk signals are present:
- Clean working tree
- Paths touched are entirely under known safe directories (e.g. `docs/`, `tests/fixtures/`)
- No recently failed CI
- No near tags (within `auto_full_when_tag_within_commits` distance)
- No policy file flags triggered

This enforces a fail-closed traceability model: *we have to see the work look harmless before we trust that it is.*

### 1.2 Threat Model

The kickoff + trace system defends against specific adversarial behaviors. The defense is a **coupled mechanism** — kickoff captures the snapshot, trace catches the drift, the ledger remembers both. Either piece alone is bypassable.

| Threat | Defense | Mechanism |
|--------|---------|-----------|
| **T1: Agent-claimed intent influencing mode selection at kickoff** | Auto router's exclusion of `agent_claimed_intent` from gating signals | Router evaluates only disk-observable signals (1–7). Intent is recorded as a witness statement, never as evidence. See §4 signal 8. |
| **T2: Agent acting outside declared mode after kickoff** | `forge0 trace` reporting `SCOPE_EXCEEDED_MODE` at trace time | Trace compares actual paths touched (from `git status`/`git diff`) against the mode envelope recorded in the dump. See §7. |
| **T3: Silent policy relaxation mid-session** | `policy_sha256` recorded immutably in the dump at kickoff time | Trace reports `MODE_MISMATCH` if the current policy hash differs from the kickoff-time hash. See §7. |

**Explicit non-defense:** An agent that skips kickoff entirely, or whose session never has `forge0 trace` run on it, can evade detection. The bypass mitigations (§10) reduce but do not eliminate this surface. ForgeZero's honesty principle requires naming this gap rather than hiding it.

### 1.3 Position in Lifecycle

```text
status → doctor → kickoff → verify → receipt → ledger → trace
observe → recover → snapshot → enforce → attest → remember → audit
```

Kickoff is the session-start observability primitive. Trace is the session-end audit primitive. Together they bookend the agent interaction.

## 2. CLI Interface

### `forge0 kickoff`

```text
forge0 kickoff [--mode <auto|full|minimal>] [--explain] [--intent "..."] [--out <path>] [--json]
```

| Flag | Default | Description |
|------|---------|-------------|
| `--mode <m>` | `auto` | Force a specific mode. `auto` uses the fail-closed router. |
| `--explain` | off | Run the full router evaluation and emit the complete `ModeDecision` JSON to stdout, regardless of any `--mode` override. The override appears in the output as `selected_by: "cli_override"` so the operator sees both what the router would have chosen and what the override would force. No dump file is written, no ledger event recorded. |
| `--intent "..."` | none | Agent-claimed intent string. Recorded in `mode_decision.evidence` but never used as a deciding signal. Can also be provided via `FORGE0_INTENT` env var. |
| `--out <path>` | `.forge0/sessions/<uuid>.json` | Override the dump output path. |
| `--json` | off | Emit full `KickoffResult` JSON to stdout instead of formatted text. |

**Exit codes:**
- `0` — kickoff completed successfully
- `1` — fatal error (policy parse failure, filesystem error)

### `forge0 trace <session-id>`

```text
forge0 trace <session-id> [--json]
```

| Flag | Default | Description |
|------|---------|-------------|
| `--json` | off | Emit `TraceResult` JSON to stdout instead of formatted text. |

**Exit codes:** See §7 (Trace Tags).

## 3. JSON Schemas

### 3.1 `KickoffDump` (written to `.forge0/sessions/<uuid>.json`)

```typescript
interface KickoffDump {
  schema_version: 1;
  session_id: string;       // UUID v4, generated by ForgeZero
  timestamp: string;        // ISO 8601
  mode: 'full' | 'minimal'; // resolved mode (never 'auto' — auto resolves to one of these)

  mode_decision: ModeDecision;

  local: {
    skill_registry_merkle: RegistryMerkle;  // ALWAYS present in both modes
    repo_root: string;
    branch?: string;
    head?: string;
    dirty: boolean;
    tags_at_head: string[];
  };

  pending_agent_dump?: boolean;  // true when full mode dump is written before agent response

  honesty: {
    claim: string;
    verified: string[];
    notObservable: string[];
  };
}
```

### 3.2 Agent Dump (written by the agent to `.forge0/sessions/<uuid>.agent.json`)

In full mode, the agent writes a separate file following the kickoff prompt instructions (§8). ForgeZero reads both files and synthesizes a combined view in memory when needed (e.g., when `forge0 trace` runs); the on-disk records remain immutable. The witness/observation distinction is preserved by keeping the two files separate.

```typescript
interface AgentDump {
  schema_version: 1;
  session_id: string;                         // must match the kickoff dump's session_id
  timestamp_iso: string;                      // agent-reported timestamp
  agent_self_id: string;                      // model identifier as the agent understands it
  model: string;                              // model version
  conversation_id: string;                    // conversation ID if available, else empty string
  available_skills_block: string;             // raw UTF-8 text of the <available_skills> block
  available_skills_block_sha256: string;      // SHA-256 of the raw block bytes, lowercase hex
  skill_slugs: string[];                      // extracted skill slugs, sorted lexicographically
  honesty_notes: string;                      // partial visibility caveats, empty string if none
}
```

### 3.3 `ModeDecision`

```typescript
interface ModeDecision {
  selected: 'full' | 'minimal';
  selected_by: 'auto' | 'cli_override' | 'policy_default';
  cli_override_rejected?: boolean;  // true when --mode minimal was requested but observables forced full
  evidence: ModeSignal[];
  policy_path: string | null;
  policy_sha256: string | null;
}
```

### 3.4 `ModeSignal`

```typescript
interface ModeSignal {
  signal: string;            // e.g. 'branch', 'paths_touched', 'near_tag', 'agent_claimed_intent'
  value: unknown;            // the observed value
  verdict: 'low_risk' | 'high_risk' | 'neutral' | 'not_observable';
  rule?: string;             // which policy field drove the verdict
}
```

### 3.5 `Policy` (`.forge0/policy.json`)

```typescript
interface Policy {
  default_mode: 'auto' | 'full' | 'minimal';
  default_branch_ref: string;                       // diff base for committed changes (default: 'origin/main')
  auto_full_paths: string[];                        // glob patterns; touched paths matching ANY → force full
  auto_full_branches: string[];                     // glob patterns; branch matching ANY → force full
  auto_full_when_tag_within_commits: number;        // if HEAD is within N commits of a tag → force full
  auto_full_when_uncommitted_in_paths: string[];    // glob patterns for uncommitted changes
  minimal_allowed_paths: string[];                  // globs; ALL touched paths must match these for minimal
  minimal_max_lines_changed: number;                // if lines changed exceeds this → force full
}
```

### 3.6 `RegistryMerkle`

```typescript
interface RegistryMerkle {
  slug_hashes: Record<string, string>;  // { "skill-name": "sha256-of-SKILL.md-content" }
  set_hash: string;                     // sha256 of canonical JSON of slug_hashes (sorted keys)
  computed_at: string;                   // ISO 8601
}
```

### 3.7 `KickoffResult` (returned by `runKickoff()`)

```typescript
interface KickoffResult {
  success: boolean;
  session_id: string;
  mode: 'full' | 'minimal';
  dump_path: string;
  mode_decision: ModeDecision;
  registry_merkle: RegistryMerkle;
  honesty: {
    claim: string;
    verified: string[];
    notObservable: string[];
  };
}
```

## 4. Mode Signal Enumeration

The auto-mode router evaluates these signals, in order:

| # | Signal | Source | Observable | Low-risk criteria |
|---|--------|--------|------------|-------------------|
| 1 | `branch` | `git branch --show-current` | Branch name string | Does NOT match any `auto_full_branches` glob |
| 2 | `paths_touched` | `git diff --name-only $(git merge-base HEAD <default_branch_ref>)` ∪ `git status --porcelain` | Array of file paths | ALL paths match at least one `minimal_allowed_paths` glob AND no path matches any `auto_full_paths` glob. **Note:** Empty paths array → `low_risk` (vacuously satisfies both conjuncts). This is the intended behavior for clean trees at session start. |
| 3 | `uncommitted_paths` | `git status --porcelain` | Array of modified/untracked paths | No path matches any `auto_full_when_uncommitted_in_paths` glob |
| 4 | `near_tag` | `git describe --tags --abbrev=0` → distance via `git rev-list --count <tag>..HEAD` | Commit distance to nearest tag | Distance > `auto_full_when_tag_within_commits` |
| 5 | `lines_changed` | `git diff --numstat $(git merge-base HEAD <default_branch_ref>)` sum + uncommitted numstat | Integer | Total lines changed ≤ `minimal_max_lines_changed` |
| 6 | `working_tree_clean` | `git status --porcelain` output empty | Boolean | Tree is clean (no uncommitted changes) |
| 7 | `recent_ci_failure` | `gh run list --limit 5 --json conclusion` (best-effort) | Boolean | No recent failures. If `gh` unavailable → `not_observable`, treated as `neutral` (does not force full by itself, but prevents downgrade if other signals are borderline) |
| 8 | `agent_claimed_intent` | Passed via `--intent "..."` CLI flag or `FORGE0_INTENT` env var | String | Consistent with observables. NEVER the sole deciding signal. Recorded in evidence for audit trail. |

The diff base for committed changes (signals 2 and 5) is `git merge-base HEAD <default_branch_ref>`, where `default_branch_ref` is configurable via `policy.default_branch_ref` (default: `origin/main`, falling back to `origin/master`). This captures all changes on the current branch since divergence, unioned with uncommitted changes from `git status --porcelain`.

**Decision algorithm:** Start at `full`. Downgrade to `minimal` only if signals 1–6 all return `low_risk` AND signal 7 returns `low_risk` or `not_observable` AND no `auto_full_*` rule fires. Signal 8 is recorded but does not participate in the downgrade decision — it is a witness statement, not evidence.

## 5. Policy File

Lives at `.forge0/policy.json`, version-controlled with the repository.

### 5.1 Default Policy (when file is absent)

```json
{
  "default_mode": "auto",
  "default_branch_ref": "origin/main",
  "auto_full_paths": [
    "src/**", ".github/**", "bin/**",
    "*.config.js", "*.config.ts",
    "package.json", "package-lock.json",
    "tsconfig.json"
  ],
  "auto_full_branches": [
    "main", "master", "release/*", "hotfix/*"
  ],
  "auto_full_when_tag_within_commits": 5,
  "auto_full_when_uncommitted_in_paths": ["src/**", "bin/**"],
  "minimal_allowed_paths": [
    "docs/**", "*.md", "tests/fixtures/**",
    ".forge0/**", "LICENSE", ".gitignore"
  ],
  "minimal_max_lines_changed": 50
}
```

This default errs toward more capture; for projects where most `src/` work is routine, tune `minimal_allowed_paths` and narrow `auto_full_paths` to the subset of source genuinely security-relevant for the project.

### 5.2 Policy Validation

`loadPolicy()` performs hand-written validation for each field:

| Field | Type check | Constraint |
|-------|-----------|------------|
| `default_mode` | `string` | Must be one of `'auto'`, `'full'`, `'minimal'` |
| `default_branch_ref` | `string` | Non-empty string. Should resolve to a valid git ref; non-resolution falls back to `origin/master` per §8 error handling. |
| `auto_full_paths` | `string[]` | Each element must be a non-empty string. Empty array is valid. |
| `auto_full_branches` | `string[]` | Each element must be a non-empty string. Empty array is valid. |
| `auto_full_when_tag_within_commits` | `number` | Integer ≥ 0 |
| `auto_full_when_uncommitted_in_paths` | `string[]` | Each element must be a non-empty string. Empty array is valid. |
| `minimal_allowed_paths` | `string[]` | Each element must be a non-empty string. Empty array is valid (effectively disables minimal). |
| `minimal_max_lines_changed` | `number` | Integer ≥ 0. Value of 0 means any change forces full. |

On validation failure, `loadPolicy()` throws with a message citing the file path, the field name, the expected type, and the actual value. Missing fields are filled from `defaultPolicy()` and an honesty bound entry notes which fields used defaults. Unknown fields are silently ignored (forward-compatible).

### 5.3 Policy Hash

The SHA-256 of the raw policy file content (or `null` if default policy is used) is recorded in `mode_decision.policy_sha256`. A mid-session policy edit creates a detectable fingerprint: the ledger records the policy hash at kickoff time, so post-hoc relaxation is visible.

### 5.4 Policy-Laundering Detection

`forge0 verify --mode release` should refuse authorization if `.forge0/policy.json` was modified in the same commit range as production code without a separate dedicated commit. (This is a future enhancement — noted here for architectural awareness, not implemented in v0.2.0 Phase 5.)

## 6. Universal Merkle Baseline

Both `minimal` and `full` modes ALWAYS capture `local.skill_registry_merkle`:

1. Enumerate all subdirectories of `~/.gemini/antigravity/skills/`
2. For each directory containing `SKILL.md`, compute `sha256(content)`
3. Build `slug_hashes: { [dirname]: hash }`
4. Sort keys lexicographically
5. Compute `set_hash = sha256(JSON.stringify(sorted_slug_hashes))`

This costs <1ms on typical registries and enables drift detection across sessions even without a full dump.

If the skills directory is unreadable, the merkle block is still written with `slug_hashes: {}`, `set_hash: sha256("{}")`, and an honesty bound entry noting `"skill registry not readable at <path>"`.

## 7. Trace Tags

`forge0 trace <session-id>` evaluates the kickoff dump (if present) against current disk state and produces one of these tags:

| Tag | Exit | Meaning | Trigger |
|-----|------|---------|---------|
| `NOT_OBSERVED` | 0 | No dump exists for this session | `.forge0/sessions/<id>.json` missing |
| `CORROBORATED_VIA_DISK` | 0 | Full-mode dump matches current disk state | Full dump, agent block present, all hashes match |
| `DETECTED_VIA_AGENT_REPORT` | 0 | Minimal-mode dump exists; lower confidence | Minimal dump, local merkle matches |
| `DIVERGENT_FROM_DISK` | 2 | Agent-reported slugs differ from disk | Slug set mismatch; payload lists `agent_only`, `local_only`, `common` |
| `SCOPE_EXCEEDED_MODE` | 2 | Session touched paths outside its mode envelope | Current `git diff`/`git status` paths violate the minimal-mode constraints from mode_decision. This is the drift detector for T2 (agent acting outside declared mode). |
| `MODE_MISMATCH` | 2 | Environment shifted under the session | Fires when: (a) `policy_sha256` no longer matches current `.forge0/policy.json`, OR (b) registry merkle `set_hash` changed since kickoff, OR (c) `agent_self_id` in the agent dump differs from what was expected. This is the environment-shift detector for T3, distinct from `SCOPE_EXCEEDED_MODE` which detects the agent reaching outside its declared envelope. |
| `AGENT_DUMP_PARTIAL` | 0 | Full-mode dump exists but agent block is incomplete | The `.forge0/sessions/<uuid>.agent.json` file is missing on disk, OR present but with `honesty_notes` containing partial-visibility phrases (e.g., 'cannot see', 'partial', 'restricted'). |
| `TRACE_INTEGRITY_FAILURE` | 1 | Dump is malformed, schema_version unknown, or hash mismatch | JSON parse error, unknown `schema_version`, hash recompute fails |

Every `TraceResult` carries:
```typescript
interface TraceResult {
  session_id: string;
  tag: TraceTag;
  exit_code: number;
  detail: string;
  payload?: Record<string, unknown>;
  honesty: {
    claim: string;
    verified: string[];
    notObservable: string[];
  };
}
```

## 8. Kickoff Prompt Template

When `forge0 kickoff` runs in full mode (not `--explain`), it emits the following prompt verbatim to stdout, with `<UUID>` substituted at runtime by ForgeZero. In minimal mode, no prompt is emitted.

```text
[FORGEZERO KICKOFF — session <UUID>]

Before any other work in this session, perform the following provenance dump.
This is a one-time setup step; the user has already verified its content.

Output path:    .forge0/sessions/<UUID>.agent.json
Schema version: 1

Steps, in order:

1. Read the literal text of the <available_skills> block (or whichever
   skill-availability section was injected at session start). If you only see
   partial content, record that in honesty_notes — do NOT fabricate.

2. Using a tool (bash sha256sum, Node crypto, Python hashlib — not estimation),
   compute the SHA-256 of the raw UTF-8 bytes of that block. Lowercase hex.

3. Parse the block to extract skill slugs (the <name>X</name> values, or
   equivalent). If the block uses a different format, leave skill_slugs empty
   and preserve the raw text only.

4. Write this JSON to .forge0/sessions/<UUID>.agent.tmp, then rename to
   .forge0/sessions/<UUID>.agent.json. Atomic — do not skip the rename.

   {
     "schema_version": 1,
     "session_id": "<UUID>",
     "timestamp_iso": "<current UTC ISO-8601, second precision>",
     "agent_self_id": "<model identifier as you understand it>",
     "model": "<model version>",
     "conversation_id": "<conversation ID if available, else empty string>",
     "available_skills_block": "<step 1 output, JSON-escaped>",
     "available_skills_block_sha256": "<step 2 hex>",
     "skill_slugs": [<step 3 slugs, sorted lexicographically>],
     "honesty_notes": "<partial visibility, restrictions, anything you cannot
                      verbatim-quote — empty string if none>"
   }

5. Reply with exactly one line first: "ForgeZero kickoff dump written: <UUID>".
   Then proceed with the user's request.

Constraints:
- BEFORE any other tool calls. Not after partial work.
- The hash input is the raw block bytes from step 1, not the JSON-escaped form.
- If .forge0/sessions/ does not exist, mkdir -p it. If you cannot write at
  all, say so explicitly and STOP. Do not improvise an alternate path.
- If your <available_skills> block is not visible to you, write the file with
  an empty block and honesty_notes explaining the limitation. Do not list
  skills from memory.

[END KICKOFF]
```

### 8.1 Agent Dump Delivery

The agent writes to `.forge0/sessions/<uuid>.agent.json` (atomic via tmp+rename, same protocol as §9). ForgeZero reads the agent dump alongside the main dump when constructing trace results; both files remain immutable on disk. The `pending_agent_dump: true` field in the main dump is informational (records that the kickoff expected an agent supplement) rather than a mutable state flag; trace determines agent presence by checking for `.forge0/sessions/<uuid>.agent.json` on disk.

The prompt is informational. ForgeZero does not block on receiving the agent response.

## 9. Atomic Dump Writes

Dumps are written atomically:
1. Write to `.forge0/sessions/<uuid>.tmp`
2. `fs.renameSync()` to `.forge0/sessions/<uuid>.json`

Agent dumps follow the same protocol:
1. Write to `.forge0/sessions/<uuid>.agent.tmp`
2. Rename to `.forge0/sessions/<uuid>.agent.json`

A crash between steps 1 and 2 leaves a `.tmp` file that `forge0 trace` ignores (it only looks for `.json`). A crashed kickoff never produces a partial `.json` dump.

## 10. Bypass Mitigations

Three layered mitigations prevent kickoff bypass:

1. **Minimal default friction**: When the auto router resolves to `minimal`, the kickoff is near-instantaneous and writes only a tiny JSON dump. Most routine sessions pay negligible cost.

2. **Graceful degradation**: `forge0 trace <id>` returns `NOT_OBSERVED` (exit 0) rather than crashing when no dump exists. The tool remains useful without kickoff.

3. **Soft warning**: Any `forge0` subcommand invoked with `FORGE0_SESSION_ID` env var set, where `.forge0/sessions/<id>.json` does not exist, emits a single stderr line:
   ```text
   forge0: warning: kickoff dump not found for session <id>
   ```
   This does not block execution.

## 11. Mid-Session Scope Drift

If a `minimal`-mode session drifts into touching `src/auth/` or other high-risk paths:

1. The mode is **NOT** upgraded mid-flight (which would produce ugly partial provenance)
2. The drift is detected at trace time as `SCOPE_EXCEEDED_MODE` (exit 2)
3. The payload lists which paths exceeded the minimal-mode envelope
4. The session is recoverable — the historical record is honest about what was and wasn't captured

## 12. Invariants

1. Every public-facing claim corresponds to a reproducible artifact on disk
2. The auto router is fail-closed: `full` is the default, `minimal` must be earned
3. `agent_claimed_intent` is never the sole or deciding driver of mode selection
4. Mode decision is fully auditable and reproducible from recorded evidence
5. Universal merkle baseline is always captured regardless of mode
6. Dump writes are atomic (tmp + rename)
7. No mid-session mode upgrades
8. Ledger chain format unchanged; new event kinds are additive
9. All new code is additive to the public API
10. `schema_version: 1` enables forward-compatible evolution
11. The defense against downgrade attacks is a coupled mechanism (kickoff + trace + ledger), not a single gate

## HYGIENE-FORGE-011 — dual-pattern git ls-remote command not regression-tested at runVerify level

*(No on-disk doc citation — documented from v0.2.2 dispatch)*

**Problem Statement:**
"The fix for HYGIENE-008 (annotated tag dereferencing) had two parts: (a) parseLsRemoteHash was extended to prefer the peeled ^{} SHA, and (b) the execSync command in runVerify was changed from git ls-remote --tags origin ${tagName} to git ls-remote --tags origin "${tagName}" "${tagName}^{}" because git only emits the peeled line when pattern matching is in play. The parser change has a unit test (parseLsRemoteHash returns the peeled commit SHA for annotated tag output) but the *command* change has no test. If a future refactor reverts the dual-pattern command syntax, the parser test would still pass while the actual remote tag check would silently break for annotated tags."

**Source Extracts:**
`g:\suno\forgezero\src\verify\verify.ts`:
Line 210 (inside `runVerify` remote checks):
```typescript
          const out = execSync(`git ls-remote --tags origin "${tagName}" "${tagName}^{}"`, { cwd: repoRoot, encoding: 'utf-8', stdio: ['pipe', 'pipe', 'ignore'] });
```

*(No on-disk doc citation — documented from v0.2.2 dispatch)*

**Problem Statement:**
"The LedgerEventKind union in src/ledger/types.ts includes values that may not all be reachable via the forge0 ledger record CLI's --event <event> flag. Currently the CLI accepts verify, receipt, and manual (added in v0.2.1 as HYGIENE-009). However, the ledger module also has internal recorders for kickoff and trace events (called automatically when forge0 kickoff and forge0 trace run), and the type system may permit other event kinds as well. The risk is that future event kinds get added to LedgerEventKind without a corresponding CLI path — this is exactly how HYGIENE-009 happened (the manual kind was already in the union but had no CLI surface). The fix is a test or type-level constraint that enforces parity between the type union and the CLI's accepted set."

## Test Coverage & Integration Map

### HYGIENE-FORGE-006 (kickoff ledger summary UUID truncation)
1. **Existing tests that touch affected code:**
   - None. There are no tests in `tests/ledger.test.ts` (or elsewhere) that execute `recordKickoffEvent` or `recordTraceEvent`.
2. **Cross-cutting code:**
   - Similar truncation exists for trace events (`recordTraceEvent` at `src/ledger/ledger.ts:390`). The fix will need to be symmetrically applied to both event types to maintain pattern consistency. Tests should be added to `tests/ledger.test.ts` to assert the non-truncated form.

### HYGIENE-FORGE-007 (pre-existing `any` casts at bin/forge0.ts)
1. **Existing tests that touch affected code:**
   - None. `tests/ledger.test.ts` only tests `forge0 ledger verify` and `forge0 ledger list` via CLI. There are no tests that exercise `forge0 verify` or `forge0 ledger record --event verify`.
2. **Cross-cutting code:**
   - The actual type `opts.mode` should be is `VerifyMode` (defined as `'precommit' | 'release' | 'bundle'` in `src/verify/types.ts`).
   - If `opts.mode` is correctly validated and narrowed at runtime, `runVerify` and `recordVerifyEvent` will enforce type safety without casts.
   - Commander returns strings by default. A common pattern is to validate against an array of allowed constants (e.g. `const validModes = ['precommit', 'release', 'bundle'] as const;`) and handle rejection explicitly, removing the need for the `as any` cast.

### HYGIENE-FORGE-011 (dual-pattern git ls-remote command not regression-tested)
1. **Existing tests that touch affected code:**
   - `tests/verify.test.ts:8` -> `it('parseLsRemoteHash handles valid output', ...)`
   - `tests/verify.test.ts:13` -> `it('parseLsRemoteHash returns null on empty/invalid', ...)`
   - `tests/verify.test.ts:19` -> `it('parseLsRemoteHash returns the peeled commit SHA for annotated tag output', ...)`
   - *Note: There are no existing end-to-end tests calling `runVerify` with `--remote` (`opts.remote = true`) against a temporary git repo.*
2. **Cross-cutting code / Integration Map:**
   - The `createTempGitRepo` helper signature: `function createTempGitRepo(version = '0.0.1'): string` (located in `tests/ledger.test.ts:27`).
   - Constraints on extending `createTempGitRepo`: It currently creates a local barebones repository with one initial commit. It does not create annotated tags or establish a remote (`origin`). To effectively test the `--remote` flag in integration, the helper (or a specialized one) would need to run `git init --bare` to simulate a remote, add it via `git remote add origin`, and push an annotated tag to it.
   - Vitest mocking: There is no existing pattern for `vi.mock` or `vi.spyOn` for `node:child_process` `execSync` anywhere in the `tests/` directory. The codebase favors execution of real child processes against temporary file structures or repositories over unit-level mocking.

### HYGIENE-FORGE-012 (CLI coverage parity for LedgerEventKind)
1. **Existing tests that touch affected code:**
   - `tests/ledger.test.ts:234` -> `it('recordVerifyEvent records a verify event in temp repo', ...)` (exercises `verify` event)
   - `tests/ledger.test.ts:240` -> `it('recordReceiptEvent records a receipt event in temp repo', ...)` (exercises `receipt` event)
   - `tests/ledger.test.ts:279` -> `it('recordManualEvent records a manual event in temp repo', ...)` (exercises `manual` event)
2. **LedgerEventKind values and creation paths:**
   - `verify`: Created via CLI directly (`forge0 ledger record --event verify`), dispatched to `recordVerifyEvent`.
   - `receipt`: Created via CLI directly (`forge0 ledger record --event receipt`), dispatched to `recordReceiptEvent`.
   - `manual`: Created via CLI directly (`forge0 ledger record --event manual`), dispatched to `recordManualEvent`.
   - `kickoff`: Created via automatic recorder (`recordKickoffEvent`), called during `forge0 kickoff`.
   - `trace`: Created via automatic recorder (`recordTraceEvent`), called during `forge0 trace`.
   - `doctor`: Unreachable (no CLI surface or internal recorder implementation).
   - `status`: Unreachable (no CLI surface or internal recorder implementation).
   - `hook-install`: Unreachable (no CLI surface or internal recorder implementation).
   - `bundle`: Unreachable (no CLI surface or internal recorder implementation).
3. **Parity test feasibility:**
   - Defining full parity strictly at the TypeScript static level is complex because `LedgerEventKind` is a type union and cannot be iterated at runtime. A "parity test" is possible by redefining the canonical list of accepted CLI values as a runtime tuple/array (`const CLI_ACCEPTED_EVENTS = [...] as const;`), exporting it from `forge0.ts` or `types.ts`, and asserting in vitest that the type derived from it matches expected union bounds, or by iterating over the array to ensure the CLI's `if/else` router handles every member. 

### Naming Conventions and Severity Levels
- Severity levels are defined in `LedgerCheck` as `'info' | 'low' | 'medium' | 'high' | 'critical'`.
- Verification modes map strictly to `'precommit' | 'release' | 'bundle'`.
- Artifact follows the structure set out by the `v0.2.1-hygiene-recon-supplement.md` standard.
